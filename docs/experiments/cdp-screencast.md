# Experiment: CDP screencast stream pane

**Status:** Draft / proof-of-concept · **Branch:** `cdp-screencast-experiment` · **Owner:** TBD

## Summary

The existing **Browser** pane in Band only worked in the desktop app —
tabs are Electron `WebContentsView` instances and the web build showed a
fallback. There was no way to *see* the browser tabs you opened on the
desktop when connecting to the same Band server from another device.

This experiment fills the gap, but the more important contribution is
**reframing the browser tab as workspace state instead of UI state**:
tabs live in the DB; both desktop and web are clients that observe and
drive them. The web's Browser pane is a screencast viewer over CDP; the
desktop's Browser pane is the existing native webview overlay; both
target the same chromium target, so they don't diverge.

## How it works

```
   ┌────────────────────────┐                       ┌─────────────────────────┐
   │  Band web frontend     │                       │  Band desktop           │
   │  ScreencastPanel       │                       │  (Electron)             │
   │   - GET /api/cdp/tabs  │                       │                         │
   │     (workspaceId)      │                       │  React tree:            │
   │   - WS /cdp?bandTabId  │                       │   BrowserHostBridge     │
   └────────┬───────────────┘                       │     (subscribes to      │
            │                                       │      browserHost.       │
            ▼                                       │      ensureView)        │
   ┌────────────────────────┐                       │                         │
   │  Band web server       │  tRPC subscription    │  Main process:          │
   │   /api/cdp/tabs:       ├──── ensureView ──────▶│   ipcMain.handle(       │
   │     listBrowsers(ws)   │                       │     browser_ensure)     │
   │   /api/cdp/snapshot/X: │  tRPC mutation        │   ipcMain.handle(       │
   │     captureSnapshot(X) │◀── targetReady ───────┤     browser_get_cdp_    │
   │   /cdp ws proxy:       │  tRPC mutation        │     target)             │
   │     ensureCdpTargetId  │◀── viewDestroyed ─────┤                         │
   │     → upstream ws to   │                       │  BrowserViewManager:    │
   │     :9223/devtools/    │                       │   .ensure(args)         │
   │     page/<cdpTargetId> │                       │   .getCdpTargetId(args) │
   └────────────────────────┘                       │   .destroy(args) ───┐   │
                                                    │                     │   │
                                                    │   emit              │   │
                                                    │   browser-view-     │   │
                                                    │   destroyed ────────┘   │
                                                    └─────────────────────────┘
                                                            │
                                                            ▼
                                          chromium :9223  /json/list  /devtools/page/X
```

The web client only ever speaks in **bandTabId** (Band's persistent
`browser_<uuid>`, stored in `panel_states`). The web server resolves
each bandTabId to a transient **cdpTargetId** (chromium's per-page UUID)
on demand and proxies CDP traffic through `:9223`. The mapping is
populated lazily via the `browserHost.ensureView` tRPC subscription:
when the desktop's `BrowserHostBridge` receives an event it calls the
local `browser_ensure` and `browser_get_cdp_target` IPC handlers, then
reports the cdpTargetId back via the `targetReady` mutation. On view
destroy (LRU eviction, explicit close, app quit), the desktop emits the
`browser-view-destroyed` IPC event, which the bridge forwards to the
`viewDestroyed` mutation so the server clears its cache.

### Identifier model

| | `bandTabId` | `cdpTargetId` |
|---|---|---|
| Example | `browser_a3b20c2f-…` | `A3B20C2F788D06B932AC7D3510B4E128` |
| Generated by | Band, on `browsers.create` | Chromium, on `webContents` construction |
| Lives in | `panel_states.id` (DB) | chromium process memory |
| Survives desktop restart? | Yes | No — chromium exits, all targets gone |
| Survives LRU eviction? | Yes | No |
| Visible to | UI clients, agents, tRPC | CDP protocol only |

The same `bandTabId` will map to a different `cdpTargetId` every time
the desktop restarts or the WebContentsView is LRU-evicted. The cache
in `apps/web/src/lib/browser-host.ts` is the only place the mapping is
materialised, and it's always rebuildable from the DB plus a fresh
ensure roundtrip.

## What works

- ✅ Single-adapter architecture: the desktop is the canonical chromium
  host. Web client streams through the Band server, which proxies to the
  desktop's `--remote-debugging-port=9223` endpoint.
- ✅ Layout matches the desktop Browser pane: `DockviewReact` with the
  same `dockview-theme-band dockview-browser-tabs` theme. Each Band
  browser tab is a panel with an address bar above the live stream.
- ✅ Tab list comes from Band's DB (`browsers.list`), exposed via
  `/api/cdp/tabs?workspaceId=X`. No URL-based filtering, no
  `document.visibilityState` polling — those were artifacts of the old
  "filter `/json/list`" approach.
- ✅ Lazy view materialisation: when the web client opens a stream for a
  Band tab whose `WebContentsView` doesn't exist yet on the desktop
  (e.g. the desktop's dockview hasn't placed the panel), the
  `browserHost.ensureView` bridge spawns one. The web sees a brief
  "Connecting…" before the first frame arrives.
- ✅ Within-session persistence: the same `WebContentsView` survives
  workspace switches and panel unmounts (existing `BrowserViewManager`
  behaviour, LRU cap of 10). Closing the web pane and re-opening on
  desktop reveals the same instance with DOM, scroll, and JS state
  preserved.
- ✅ Cache invalidation on view destroy: the desktop's emit on `destroy`
  → tRPC mutation → server cache clears → next stream attempt triggers
  a fresh ensure. No more stale-targetId-after-LRU-eviction failures.
- ✅ Mouse + scroll forwarding: `Input.dispatchMouseEvent` for click,
  drag, move, wheel.
- ✅ Friendly fallback: when no desktop is connected, `/api/cdp/tabs`
  returns `error: "Open the Band desktop app…"` and the web pane shows
  it.

## What's intentionally not in scope

- ❌ **Standalone Chromium fallback.** The plan called this out as
  follow-up. Today, no desktop = no streaming. Adding a server-spawned
  chromium for headless / agent use is the next iteration.
- ❌ **Agent CDP tool surface.** No `capture-element`, `query-DOM`, etc.
  yet. The foundation gives agents a stable bandTabId-keyed target;
  building tools on top is a small additional layer.
- ❌ **Element-picker UX.** Web user clicks a streamed pixel → server
  resolves to a DOM node and selector. Trivial to add later via
  `DOM.getNodeForLocation`.
- ❌ **Desktop renderer rewrite.** Desktop still uses native
  `WebContentsView` for the active tab — same path as before. The
  eventual fix (desktop also streams from server, single rendering
  path) is acknowledged but not pursued.
- ❌ **Cookie / session sharing.** Cookies live inside the chromium
  target. Across desktop restarts the target is fresh, so logins reset.
  Same behaviour as a real browser between restarts.
- ❌ **Tab cap / hibernation.** Existing `MAX_BROWSER_VIEWS = 10` LRU
  applies. Long-term: configurable cap, idle eviction, soft hibernation.

## Files

**New:**
- `apps/web/src/lib/browser-host.ts` — bandTabId↔cdpTargetId cache,
  ensure protocol, subscriber list.
- `apps/web/src/components/BrowserHostBridge.tsx` — desktop-only React
  component that translates ensureView events to IPC and reports back.

**Edited:**
- `apps/web/src/lib/cdp-targets.ts` — `captureSnapshot` takes a
  bandTabId; URL filter / visibility checks deleted.
- `apps/web/src/lib/cdp-proxy.ts` — `/cdp?bandTabId=…` proxy resolves
  via `ensureCdpTargetId`.
- `apps/web/start-server.ts` + `apps/web/vite.config.ts` —
  `/api/cdp/tabs?workspaceId=…` reads from DB; `/api/cdp/snapshot/<id>`
  is the new snapshot endpoint; old active-tab routes deleted.
- `apps/web/src/trpc/router.ts` — adds `browserHost` router with
  `ensureView` subscription, `targetReady` + `viewDestroyed` mutations.
- `apps/web/src/components/ScreencastPanel.tsx` — drops click-cooldown
  + visibility polling + Page.bringToFront; data source is
  `/api/cdp/tabs?workspaceId=…`; WS uses `bandTabId`.
- `apps/web/src/routes/__root.tsx` — mounts `<BrowserHostBridge />`.
- `apps/desktop/src/browser/view-manager.ts` — adds `ensure(args)` and
  `getCdpTargetId(args)` methods + emits `browser-view-destroyed`.
- `apps/desktop/src/main/ipc/browser.ts` + `register.ts` — adds the
  `browser_ensure` and `browser_get_cdp_target` handlers.
- `apps/desktop/src/preload/index.cts` — allowlists the new channel and
  event names.
- `apps/desktop/src/shared/ipc-channels.ts` + `types.ts` — channel and
  payload definitions.

**Deleted:**
- `apps/web/src/lib/cdp-broadcast.ts` — SSE-based active-tab pub/sub
  from the previous design, no longer needed under single-adapter.

## Manual demo

```bash
# Run the desktop app from this branch (for the --remote-debugging-port
# switch and the new IPC handlers).
pnpm --filter @band-app/desktop dev
```

In the desktop:
- Open the Browser pane (⌘B). Create a couple of tabs.

From any other browser:
- Open the same Band server URL.
- Activate the Browser pane (⌘B). The web pane shows the same tabs.
- Click a tab → live stream starts. Click links / scroll → the desktop
  tab reacts.
- Switch workspaces in the web. Come back. Tab strip is still there,
  stream resumes.
- Close the web tab, return to desktop. Same WebContentsView, same DOM,
  same scroll position.
- Quit the desktop. Web pane shows the "Open the desktop app" message.
  Restart desktop → web reconnects, tab list returns.
